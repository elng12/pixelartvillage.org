# Claude Code 开发规范

本文档用于指导 Codex（终端版 Codex CLI 代理）在本仓库的工作方式，确保改动可控、沟通高效、风险可见。除非上级指令另有要求，本说明适用于仓库根目录及所有子目录中的文件与任务。

## 适用范围

- 作用域：本仓库根目录及全部子目录。
- 优先级：显式的系统/开发者/用户指令 > 本 AGENTS.md > 其他文档。
- 若子目录存在更深层的 AGENTS.md，与本文冲突时以更深层为准。

## 协作与沟通

- 中文沟通：所有解释、变更说明、总结均使用中文。
- 事前提示（preamble）：在运行一组相关命令前，用 1–2 句简述要做的动作与目的。
- 计划跟踪：对多步骤或有不确定性的任务，使用 `update_plan` 维护计划；始终保持最多只有一个 `in_progress` 步骤。
- 进度回报：长时操作应适度汇报（8–12 个字），概述做了什么、接下来做什么。

## 工具与沙箱

- 优先使用 `apply_patch` 修改或新增文件；不要用其他方式写入文件。
- 读取/搜索：单文件读取请分块（每次 ≤ 250 行）。全文搜索优先用 `rg`；若无 `rg`，可用 `findstr`/`Select-String` 等替代。
- 沙箱与审批：
  - 仅在需要写入工作区外、访问网络、或执行潜在破坏性操作时请求审批。
  - 若因沙箱失败阻塞关键任务，可带理由请求无沙箱执行。
- 不擅自运行 GUI/打开外部程序；危险命令（如 `rm -rf`、批量重命名/移动）须明确获批。

## 变更原则

- 最小必要改动：只改动完成任务所需的最小范围。
- 修复根因：尽可能修在根因而非表面绕过。
- 保持一致：遵循现有代码风格、目录结构、命名习惯。
- 不做无关修复：不要顺手修复与任务无关的其他问题（可在结果中指出）。
- 文件命名与移动：非必要不重命名/移动文件；若必须，需明确说明理由与影响面。
- 不添加版权/许可证头（除非明确要求）。

## 代码风格

- 总则：严格遵循项目现有的格式化与 Lint 配置；不要引入新的格式化器或风格工具。
- 命名规范：
  - 变量/函数：`lowerCamelCase`
  - 类/类型/组件：`UpperCamelCase`
  - 常量：`UPPER_SNAKE_CASE`
  - 文件：`kebab-case`（JS/TS/样式）；若项目对组件文件有 `PascalCase` 约定则遵循其约定。
- 缩进与行宽：遵从项目配置；若未配置，建议 2 空格缩进、行宽 100。
- 引号与分号：遵从项目配置；未配置时建议单引号并保留分号。
- 导入顺序：内置模块 → 第三方依赖 → 本地模块；同组内按字母序；相对路径尽量短且稳定。
- 模块与导入：
  - 使用 ES 模块语法（`import`/`export`），不要使用 CommonJS（`require`/`module.exports`）。
  - 尽量使用解构导入：`import { foo } from 'bar'`；仅在确有默认导出时使用 `import foo from 'bar'`。
  - TypeScript 中对类型使用 `import type { T } from '...'`，分离值与类型导入以优化构建。
  - 避免 `export * from '...'`；明确导出清单或使用 `export { foo } from '...'`。
  - 路径保持稳定与可维护；若项目已配置路径别名（如 TS `paths`/Vite 别名），优先使用别名而非深层相对路径。
- 可读性优先：函数尽量短小、单一职责、倾向提前返回；减少嵌套层级，必要时提取辅助函数。
- 类型（TypeScript）：优先显式类型；启用严格模式；避免 `any` 与非空断言 `!`，可用 `unknown` + 类型收窄；为公共 API 导出类型定义。
- 错误处理：不吞异常；抛错时提供上下文信息（参数、状态）；在边界处捕获并转换为业务可理解的错误。
- 注释：描述“为什么”而非“做什么”；公共 API 使用 JSDoc/TSDoc；临时欠账用 `TODO`/`FIXME` 并附原因或链接；注释语言以项目现有约定为准并保持一致。
- 日志：避免遗留调试日志；输出分级（debug/info/warn/error）；不记录敏感信息。
- 空值处理：优先使用可选链 `?.` 与空值合并 `??`；避免过度断言。
- 导出策略：倾向命名导出，控制公共 API 面；每文件聚焦单一职责。

## 测试与验证

- 优先局部验证：先针对改动点进行最小化验证，再逐步扩大范围。
- 测试策略：
  - 若仓库已有测试或可构建/运行，优先使用现有机制验证。
  - 若无对应测试，且仓库已有测试结构，可在相邻位置添加最小必要的测试；否则不要为无测试的仓库强行新增测试框架。
- 不修复与任务无关的测试失败（可在总结中提示）。

## 输出规范（面向用户的最终消息）

- 风格：简洁、直接、友好；以事实为主，避免冗长重复。
- 结构：必要时使用小节标题（1–3 个词），并使用精炼要点列表。
- 代码/命令：使用反引号包裹路径、命令、代码标识符（如 `src/index.ts`、`npm run build`）。
- 不输出 ANSI 转义码；不输出学术式内联引用标记。

## 常见任务流程

- 新功能：
  - 澄清需求 → 勘察代码结构 → 小范围实现 → 局部验证 → 扩大验证 → 产出简明交付说明。
- 缺陷修复：
  - 复现问题 → 定位根因 → 最小改动修复 → 验证修复 → 列出影响面与回归点。
- 配置/脚本调整：
  - 备份或说明变更点 → 修改 → 验证行为一致且预期生效 → 记录回滚方式。

## 运行与命令发现（语言无关）

- 在执行前，应先“探测项目类型与命令”：
  - 前端/Node：查找 `package.json` / `pnpm-lock.yaml`，识别 `scripts` 中的 `dev`/`build`/`test`。
  - Python：查找 `pyproject.toml`/`requirements.txt`/`poetry.lock`，识别 `pytest`/`unittest`/启动脚本。
  - Rust：查找 `Cargo.toml`，识别 `cargo build`/`cargo test`。
  - Go：查找 `go.mod`，识别 `go build`/`go test`。
  - 其他：根据仓库已有脚本或 README 说明决定。
- 若无法可靠识别命令，应先与用户确认，而非盲目尝试安装依赖或运行任务。

## 审批策略（建议）

- 无需审批：
  - 只读操作（查看文件、搜索、分析）。
  - 在工作区内使用 `apply_patch` 编辑文件。
- 需要审批：
  - 访问网络、安装依赖、或长时间占用资源的命令。
  - 写入工作区外、修改系统/环境级配置。
  - 潜在破坏性或不可逆的操作（删除/批量移动/覆盖）。

## 禁止或需谨慎的行为

- 不得捏造信息或虚构运行结果；无法获取的信息需主动声明并请求指引。
- 避免一次性大改动；若改动范围较大，先给出拆分计划并征求确认。
- 对第三方服务或凭据相关信息保持敏感，不向外泄露。

## 交付物要求

- 代码改动通过 `apply_patch` 提交。
- 最终消息需包含：
  - 变更摘要（做了什么、影响范围）。
  - 简要验证说明（如何验证变更有效）。
  - 后续建议或可选下一步。

## Bash命令

- 安全基线：脚本建议使用 `set -euo pipefail`（在适用环境）；仅在确需容错时局部关闭并标注理由。
- 只读优先：查看/搜索优先，分段读取单次不超过 250 行或约 10KB 输出。
- 搜索规范：全文搜索优先 `rg -n -S '关键词'`；列文件用 `rg --files`；若无 `rg` 再用 `grep`/`find` 等替代。
- 引号与分隔：对路径/变量始终加引号；使用 `--` 终止选项；含空格/特殊字符的文件名用 `find -print0 | xargs -0` 传递。
- 文件变更：一律通过 `apply_patch` 修改或新增文件；不要用 `sed -i`、`ed`、`perl -pi` 等直接改写。
- 作用范围：仅对工作区内相对路径操作；不要触碰工作区外路径；禁止 `sudo` 与系统级修改。
- 危险命令：`rm -rf`、批量 `mv`、`chmod -R`/`chown -R`、`dd` 等需明确批准；严禁使用 `rm --no-preserve-root`。
- 网络访问：`curl`/`wget`/包管理器联网操作默认禁止，需审批并说明用途与范围。
- 输出可读性：尽量关闭彩色输出（如 `--color=never`），避免输出 ANSI 转义码。
- 分页查看：`sed -n '1,200p' <path>`、`sed -n '201,400p' <path>` 等方式分块读取。
- 搜索示例：`rg -n -S "TODO|FIXME"`、`rg --files -g "**/*.ts"`。
- 并行谨慎：避免后台作业与过度并行（如 `xargs -P`），以便记录与审计清晰。
- 平台注意：在 WSL/类 Unix 使用上述 Bash 规则；Windows 原生命令遵循“工具与沙箱”章节的等效规范（如 `Select-String`/`findstr`）。

## HTML 规范

- 使用语义化HTML标签（`<header>`, `<nav>`, `<main>`, `<section>`, `<article>`等）
  - 标题层级从 `h1` 开始、按序递进，不跳级。
- 保持标签嵌套的一致缩进。
- 必要的无障碍属性（`alt`, `aria-label`, `role`等）。

## Tailwind CSS 规范

- 核心原则：以 utility-first 为主，优先组合类，不写自定义 CSS；确需扩展时在 `@layer` 中维护。
- 设计令牌：统一在 `tailwind.config.{js,ts}` 的 `theme.extend` 定义颜色、间距、字体、阴影、zIndex、容器等，避免魔法数。
- 任意值使用：仅在确需时用任意值 `[]`，随后将其沉淀回配置；能用 `theme()` 的不用硬编码。
- 类顺序与格式化：启用 `prettier-plugin-tailwindcss` 自动排序；未使用插件时遵循下述“类名顺序”。
- **类名顺序**：布局→尺寸→间距→颜色→字体→其他
- 变体顺序：`dark:` → 断点 `sm`→`md`→`lg`→`xl`→`2xl` → 交互 `hover`/`focus`/`active`/`disabled` → `aria-*`；保持全局一致。
- 响应式前缀顺序：`sm:` → `md:` → `lg:` → `xl:` → `2xl:`。
- 状态前缀在后：`hover:`、`focus:`、`active:` 等应置于断点前缀之后。
- 响应式与容器：移动优先；使用断点前缀而非手写媒体查询；使用 `container` 并在配置中开启 `center` 和默认 `padding`。
- 暗色模式：使用 `class` 模式，在根节点切换 `dark`；颜色配比通过设计令牌完成，不直接使用随意颜色。
- 复用样式：复用的样式组合提取为组件或 CSS 类；在 Tailwind 项目中优先在 `@layer components` 中使用 `@apply` 抽取到如 `.btn`/`.card` 等，避免在模板中重复冗长的类串。
 - 使用 @apply 封装：考虑使用 `@apply` 指令封装常用样式组合，并在 `@layer components` 中维护；命名清晰（如 `.btn-primary`），避免层层叠加导致难以维护。
- 避免内联样式：统一使用 Tailwind 工具类；若需动态样式，优先通过条件类名实现；确需内联时应给出理由，并尽快回收为配置/组件。
- 动态类名：避免字符串拼接构造未知类；使用 `clsx`/`classnames`/`cn` 条件组合；必要时在 `safelist` 明确保留。
- 构建与 tree-shaking：`content` 覆盖所有模板路径，避免遗漏；不要在运行时生成不可静态分析的类名。
- 插件与排版：按需启用 `@tailwindcss/forms`、`typography`、`aspect-ratio`；富文本用 `.prose` 管理排版。
- 无障碍：保留可见焦点样式；配合 `focus-visible:`、`aria-checked:` 等变体；颜色对比满足 WCAG。
- 长类名换行：超过 3–4 个工具类时可按语义分组换行，提高可读性。

示例：

```html
<div class="flex w-full p-4 bg-blue-500 text-white font-bold rounded"></div>
```

## 组件化建议

- 颗粒度：单一职责、小而专注；当组件超过 ~150 行或承担多职责时应拆分。
- 输入/输出：显式定义 `props`/参数与事件/回调，提供合理默认值与类型约束；避免隐式全局依赖。
- 状态策略：优先无状态展示组件；将业务状态上提到容器/页面层；必要的本地状态保持最小化。
- 可组合性：组合优于继承；支持插槽/children 或等价机制；必要时提供 render-prop/slots API。
- 可扩展性：透传原生属性（如 `...rest`）；暴露 `className`/`style`/`slot` 等扩展点并保持向后兼容。
- 样式约定：以 Tailwind 为主；避免写死颜色/间距，使用设计令牌；重复样式提取为复用单元或 `@apply`（在 `@layer components`）。
- 无障碍：使用语义标签与必要的 `aria-*` 属性；可键盘操作并有可见焦点；管理好焦点转移。
- 性能：避免不必要重渲染（合理切分、memo/缓存）；长列表采用虚拟化；按需/懒加载次要子组件。
- 命名与结构：组件/目录使用 `PascalCase`；每组件目录内包含实现、测试与导出入口（`index.ts`）。
- 错误边界：在上层容器捕获与兜底；组件内部抛出具备上下文信息的错误。
- 文档与示例：为公共组件提供最小使用示例与 API 注释；复杂组件可配 Story/MDX 文档。
- 测试：为关键交互、可访问性与状态边界编写单测；谨慎使用易碎快照。
- 依赖控制：尽量零依赖或轻依赖，不引入沉重库；跨组件共享逻辑抽取为工具函数或 Hook。

## 工作流程

- 需求澄清：明确目标、背景、范围、约束与验收标准；列出不确定点并向需求方确认。
- 项目勘察：浏览目录结构与关键配置（如 `README`、`package.json`、`tailwind.config` 等）；识别可用命令（如 `npm run typecheck`、`npm run build`）。
- 制定计划：使用 `update_plan` 将任务拆分为 3–7 个短步骤，标注 `pending`/`in_progress`/`completed`，始终仅有一个 `in_progress`。
- 变更实施：仅使用 `apply_patch` 修改/新增文件；遵循“最小必要改动”“保持一致”；每组相关操作前提供 1–2 句事前提示（preamble）。
- 验证与自检：
  - 局部优先：先验证改动点，再扩大范围。
  - 在完成一系列代码更改后务必进行类型检查：`npm run typecheck`。
  - 为了性能考虑，优先运行单个测试，而不是整个测试套件。
  - 运行类型检查与构建（若适用）：`npm run typecheck`、`npm run build`。
  - 若项目有测试脚本，按约定执行（涉及网络/写入临时目录时先征求审批）。
- 结果交付：在最终消息中给出变更摘要、影响范围、验证方式与后续建议；如需提交/合并，由维护者执行，我可提供建议的提交信息或 PR 描述。
- 审批与风险控制：访问网络、写入工作区外、或潜在破坏性操作前需说明理由与范围并请求审批；对不可逆操作提供回滚思路。
- 记录与对齐：若与子目录的 AGENTS.md 冲突，以更深层为准；必要时同步更新规范或备注差异。

## 质量验证规则

- 必过检查：
  - 类型检查通过：`npm run typecheck`。
  - 构建通过：`npm run build`。
  - 若配置了 Lint/格式化，需通过：如 `npm run lint`、`npm run format --check`。
- 测试策略：
  - 为了性能，优先运行受影响的单个或小集合测试；临近合并/发布再跑全量套件。
  - 新增或变更的逻辑应至少被一个测试覆盖，或命中既有测试。
- UI 与可访问性：遵循“HTML 规范”“Tailwind CSS 规范”，使用语义标签与必要的 `alt`/`aria-*`/`role`，交互元素具备可见焦点样式。
- 变更卫生：无未使用导入/变量、无 `console.log`/`debugger`、无临时代码；不新增未使用依赖；保持最小必要改动。
- 性能与体积：避免引入大体积依赖或明显性能退化；如配置了体积/性能检查（如 size-limit/Bundle 分析），需通过并记录差异。
- 安全：避免 `eval`/`new Function` 与不必要的 `innerHTML`；对外部输入进行校验/转义；外链 `target="_blank"` 搭配 `rel="noopener noreferrer"`。
- 兼容性与 API：不破坏既有公共 API；若为破坏性变更，标注 BREAKING CHANGE 并提供迁移说明。
- 文档与脚本：行为变更需更新 `README`/注释；新增命令同步到“项目命令”。
- 回滚与开关：重要改动提供回滚路径或特性开关，并在最终消息中说明使用方式。
- 验证记录：在最终消息中列出执行过的验证命令与结论（示例：`npm run typecheck` 通过、受影响测试通过、构建通过）。

## 定期质量检查

- 频率：每周、每迭代结束、发布前进行例行检查。
- 覆盖范围：
  - 类型检查：`npm run typecheck`。
  - 构建验证：`npm run build`。
  - Lint/格式化（如有）：`npm run lint`、`npm run format --check`。
  - 测试：优先运行受影响用例，关键里程碑跑全量套件。
  - TODO/FIXME 清理：`rg -n -S "TODO|FIXME"`。
  - 体积与性能（如有工具）：size-limit/Bundle 分析对比近期基线。
  - 依赖健康（需审批网络访问时再执行）：`npm outdated`、`npm audit`。
- 输出要求：列出执行命令与结论，问题清单与跟进计划，若存在体积/性能回退，附对比数据与改进计划。
- 审批说明：涉及网络或潜在破坏性操作（依赖升级、批量重构等）需预先说明范围并请求审批。

## 定期反问并要求提供证据

- 触发时机：在里程碑节点、重大改动前后、信息不完整或存在歧义时必须主动反问确认。
- 反问要点：需求范围、验收标准、预期输出、回滚策略、影响面与风险。
- 证据要求：提供可复现的命令与结论（类型检查/构建/受影响测试）、关键文件路径与行号、相关日志/截图（注意脱敏）。
- 复现信息：附最小复现步骤与环境说明（如 Node/npm/pnpm 版本、环境变量、运行目录）。
- 频率要求：每阶段至少一次；大型改动在合并/发布前必须完成一次证据核验。
- 不足处理：证据不充分时将事项标注为“待证实”，列出缺失清单并暂停“完成”声明，直至补齐。
- 优先文本：优先提供可检索文本日志；截图仅作补充，并附存放路径或链接。

建议反问（示例）：
- 代码真的按规范写了吗？有类型检查结果吗？
- Tailwind 类名顺序正确吗？有具体文件证明吗？
- ES 模块导入都改了吗？能看到具体改动吗？
- 构建真的成功了吗？有输出日志吗？

发现问题及时要求修正，确保每个完成声明都有可验证的证据。

## 安全原则

- 只协助防御性安全任务。
- 拒绝创建、修改或改进可能被恶意使用的代码。
- 不协助凭据发现或收集，包括批量爬取 SSH 密钥、浏览器 cookie 或加密货币钱包。
- 允许安全分析、检测规则、漏洞解释、防御工具和安全文档。
- 凭据与密钥：
  - 严禁在代码库中提交任何密钥/令牌/密码；使用本地 `.env` 与环境变量，勿上传。
  - 日志与错误信息中不可泄露敏感数据；必要时脱敏（如只显示后 4 位）。
  - 提交前执行敏感词扫描：如 `rg -n -S "(API_KEY|ACCESS_TOKEN|SECRET|TOKEN|password|passwd|密钥)"`。
- 网络与外部资源：
  - 默认禁止联网操作；如需访问网络（下载依赖、拉取外部资源），必须事先审批并说明用途与范围。
  - 引入第三方脚本/样式需谨慎；优先自托管，必要时使用 SRI 校验并锁定版本。
- 依赖安全：
  - 依赖最小化；避免来源不明或含可疑 `postinstall` 脚本的包；锁定版本避免漂移。
  - 允许的健康检查（需审批再执行联网）：`npm outdated`、`npm audit`，并记录结论与处置计划。
- 输入校验与输出转义：
  - 所有外部/用户输入均视为不可信；在边界进行类型与范围校验。
  - 输出到 HTML 时进行转义；避免 `innerHTML`/`dangerouslySetInnerHTML` 渲染未消毒内容；富文本使用可信消毒库。
- 代码执行与模板：
  - 禁止使用 `eval`/`new Function` 等动态执行；避免注入式字符串拼接执行命令。
  - Tailwind 类名动态构造需可静态分析；不可在运行时生成不可预测类名。
- 浏览器安全：
  - `target="_blank"` 链接必须加 `rel="noopener noreferrer"`；禁止使用 `javascript:` 协议链接。
  - 建议配置 CSP，默认禁止内联脚本；如需内联，使用 `nonce`/哈希并经评审。
- 文件系统与路径：
  - 仅在工作区内进行读写；禁止路径穿越；对外部路径操作需审批。
  - 写入前确认目标目录存在且正确，避免覆盖非目标文件。
- 日志与隐私：
  - 不记录个人敏感信息（PII）；移除 `console.log`/`debugger` 等调试语句。
- 构建产物：
  - 前端不可包含密钥；通过服务端代理或运行时安全注入获取必要信息。
- 权限最小化与审批：
  - 变更文件权限、批量删除/移动、系统级修改等需审批，并提供回滚方案与影响评估。
- 示例检查命令：
  - 搜索外链新窗口：`rg -n -S "target=\"_blank\"" -g "**/*.{html,tsx,jsx}"`
  - 搜索动态执行：`rg -n -S "(eval\(|new Function\()" -g "**/*.{js,ts,tsx}"`
  - 搜索直接 DOM 注入：`rg -n -S "(innerHTML|dangerouslySetInnerHTML)" -g "**/*.{js,ts,tsx}"`

## 代码评审检查清单

- 需求与范围：变更与需求一致，无隐性扩张；验收标准明确。
- 正确性与构建：`npm run typecheck` 通过；`npm run build` 通过；关键路径已走查。
- 风格一致性：遵循“代码风格”；使用 ES 模块（import/export），优先解构导入，避免 `export *`；导入顺序与命名一致。
- 可读性与复杂度：命名清晰，函数短小单一职责，倾向提前返回；重复逻辑抽取复用。
- API 兼容性：不破坏公共 API；如为破坏性变更，已标注并提供迁移说明。
- HTML 规范：语义化标签、缩进一致、必要的 `alt`/`aria-*`/`role`、`<html lang="zh-CN">`、外链 `rel="noopener noreferrer"`。
- Tailwind 规范：类名顺序与前缀顺序正确；状态前缀置后；避免内联样式；常用样式用 `@apply` 于 `@layer components` 抽取并复用。
- 测试：受影响用例已补充/更新并通过；优先运行单个测试验证，临近合并再跑全量。
- 性能与体积：未引入大体积依赖；`content` 覆盖完整；动态类名可静态分析；关键路径无明显退化。
- 安全：无密钥/凭据；无 `eval`/`new Function`；无未消毒 `innerHTML`；外链安全；仅在获批情况下联网。
- 文档与脚本：已同步更新 README/AGENTS/项目命令；提供回滚路径或特性开关。
- 证据齐备：文件路径与行号、类型检查/构建/测试结论、必要日志或截图均已提供。

通过标准
- 所有“必过检查”通过且“证据齐备”；无高风险未解决问题；评审意见已处理或给出明确计划。
- 证据缺失或不达标时，标注“待证实”，指出所需补充项并退回修改。

## 提交信息规范（Conventional Commits）

- 基本格式：`<type>(scope)!: <subject>`
  - `type`：`feat` | `fix` | `docs` | `style` | `refactor` | `perf` | `test` | `build` | `ci` | `chore` | `revert`
  - `scope`（可选）：模块或目录，使用 `kebab-case`，如 `ui/button`、`api`、`build`。
  - `!`（可选）：表示破坏性变更（BREAKING CHANGE）。
  - `subject`：简洁祈使句，中文为主，≤ 50 字，不以句号结尾。
- 正文（可选）：
  - 解释“为什么”和“如何做”，必要时分条列出影响点/迁移说明。
  - 破坏性变更可在正文或页脚使用 `BREAKING CHANGE: ...` 明确说明。
- 页脚（可选）：
  - 关联事项：`Closes #123`、`Refs #456`。
  - 破坏性变更：`BREAKING CHANGE: xxx`。
- 示例：
  - `feat(ui/button): 新增禁用态与加载中样式`
  - `fix(api): 处理空值导致的 500 错误`
  - `refactor(core): 提取校验逻辑以降低复杂度`
  - `docs(readme): 增加本地运行与构建说明`
  - `revert: feat(ui/button): 回滚因样式回归引起的改动`

分支命名
- `feat/<scope>-<summary>`（如 `feat/ui-button-loading`）
- `fix/<scope>-<summary>`（如 `fix/api-null-500`）
- `chore/<scope>-<summary>`、`docs/<scope>-<summary>` 等按需类推
- 使用 `kebab-case`，摘要简洁对齐提交标题含义

合并策略
- 优先使用 Squash & Merge，PR 标题需符合本规范（作为最终提交信息）。
- 保持提交原子化与可审查；禁止混入无关变更；必要时交互式 rebase 整理历史。

提交粒度与内容
- 一次提交只完成一件“最小可解释”的事；大改动拆分为若干可独立审阅的提交。
- 提交前自检：类型检查/构建/受影响测试通过；无调试日志与未使用代码；文档与脚本已同步。
- 如仓库启用了 commitlint/semantic-release，请遵循其现有配置。

- 可访问性：
  - 为图片提供 `alt`；纯装饰图片可空 `alt=""`。
  - 表单元素配对 `label for` 与控件 `id`；合理使用 `aria-*` 属性与 `role`。
  - `html` 标签声明语言：`<html lang="zh-CN">`。
  - 交互元素可聚焦且焦点样式可见。
- 基础元信息：`<meta charset="utf-8">`、`<meta name="viewport" content="width=device-width, initial-scale=1">`，提供唯一页面标题与必要的描述。
- 资源加载：
  - 样式表放在 `head`；脚本尽量使用 `type="module"` 并添加 `defer`。
  - 关键资源才 `preload`；跨域资源按需 `preconnect`/`dns-prefetch`。
  - 避免阻塞渲染的同步脚本。
- 图片与媒体：使用固定 `width`/`height` 以避免布局抖动；按需 `srcset`/`sizes` 或 `<picture>` 实现响应式；图片/iframe 默认 `loading="lazy"`，可加 `decoding="async"`。
- 安全与外链：外部链接若 `target="_blank"`，必须加 `rel="noopener noreferrer"`；避免内联脚本与危险的 `javascript:` URL。
- 结构与样式：避免内联样式；类名命名风格保持一致（可用 BEM）；属性值统一使用双引号；避免重复 `id`。
- 性能与 SEO：减少无用 DOM 与无意义标签；提供基础 Open Graph/社交元数据（如需要）；重要内容优先渲染。
