# SEO Feedback to PRD Wording Guide

状态：Ready for Review

日期：2026-03-22

用途：这份文档用于把外部评审意见翻译成适合并入正式 PRD 的措辞。目标不是重复策略本身，而是解决一个常见问题：

- 评审意见往往很有洞察，但措辞过满
- 正式 PRD 需要写得专业、稳健、可执行
- 凡是 Google 没公开确认的内容，不应写成“直接排名因子”或“算法铁律”

适用范围说明：本指南基于 2026 年 3 月 Google Search Central、Google Search Central Blog 与其他官方公开信息撰写。后续若有算法更新、结构化数据政策调整或搜索展示变化，建议继续沿用本指南的降级原则重新审视措辞。

本文档按三类给出建议：

1. 可以直接吸收进 PRD 的表述
2. 建议降级后再写进 PRD 的表述
3. 不建议直接写进 PRD 的表述

---

## 1. 可直接吸收进 PRD 的表述

以下表述与当前 Google Search 最佳实践高度一致，适合直接并入正式文档。

### 1.1 Tool-first page

建议正式写法：

> 对于 `image to pixel art` 这类工具型搜索意图，主转换页必须优先满足“立即开始使用”的需求。页面首屏应优先展示上传入口、结果反馈和核心操作，而不是把大量说明文案置于工具前方。

适合放入：

- 页面设计与 UX 要求
- 主转换页验收标准

### 1.2 页面 ownership

建议正式写法：

> 首页负责承接品牌与导航入口意图，`/converter/image-to-pixel-art/` 负责承接主词群与主商业意图。相关页面应通过标题、显著文案、内链与功能形态共同传达这一分工，避免首页与主转换页重复竞争同一 query cluster。

适合放入：

- ownership 策略
- 风险与缓解

### 1.3 自然内链与锚文本

建议正式写法：

> 首页、格式页与相关 Blog 应持续向主转换页传递上下文相关的站内链接。锚文本应保持清晰、自然，并采用主锚文本与语义变体组合，避免全站机械重复同一表达。

适合放入：

- 内链策略
- 执行规则

### 1.4 长尾相关实体覆盖

建议正式写法：

> 对 `Minecraft`、`AI` 等相关实体和扩展需求，第一阶段优先采用主转换页区块或 FAQ 的方式进行覆盖，而不立即扩张为独立页面。目标是增强语义完整度，而不是分散主页面权重。

适合放入：

- 关键词体系
- 页面执行规则

### 1.5 可抓取性

建议正式写法：

> 关键高信号内容，包括 `title`、`H1`、首段说明、FAQ 与核心 CTA，必须在 Google 可抓取、可渲染的 HTML 中稳定可见。实现方式可以是预渲染、SSR 或其他能确保稳定输出的方案。

适合放入：

- 技术 SEO
- 可抓取性验证

## 2. 建议降级后写入 PRD 的表述

这些观点本身有价值，但如果直接照搬，容易把推断写成 Google 已公开确认的规则。建议改写后再纳入。

### 2.1 Dwell Time / Pogo-sticking

不建议直接写：

> Dwell Time 是核心排名信号。

建议正式写法：

> 工具优先页面通常更有利于提升用户满意度、交互完成率和后续留存，这些信号往往与更好的搜索表现相关。

说明：

- Google 没有公开把 Dwell Time 或 Pogo-sticking 定义为直接排名因子
- 但把它们视为“用户满意度 proxy（近似信号）”是合理的

### 2.2 FAQ rich result

不建议直接写：

> 加 `FAQPage` schema 会带来 FAQ 富结果。

建议正式写法：

> `FAQPage` schema 的主要价值在于帮助搜索引擎理解页面结构与问答关系，而不是保证出现富结果展示。

说明：

- 目前 FAQ 富结果展示范围明显收紧
- 对一般工具站，不应把“拿到 FAQ 富结果”当成主要预期

### 2.3 SoftwareApplication + Free

不建议直接写：

> 标注 `price: 0` 就能拿到 Free rich result。

建议正式写法：

> `SoftwareApplication` 结构化数据有助于强化页面的工具属性与产品语义，但富结果是否展示并不受单一字段保证。

说明：

- `price`、`applicationCategory` 等字段可以帮助表达产品属性
- 但不能承诺会出现特定搜索样式

### 2.4 性能是决定性排名因子

不建议直接写：

> INP 或性能是决定性排名因子。

建议正式写法：

> 对工具型页面而言，交互性能会显著影响用户体验与行为指标，因此应将 INP、首屏可操作性和上传反馈速度纳入验收要求。

说明：

- 性能重要
- 但不应写成“单一决定因素”或绝对性的官方结论

### 2.5 SSR 是唯一正确解

不建议直接写：

> 只有 SSR 才符合 SEO。

建议正式写法：

> 无论采用 SSR、预渲染或其他渲染方案，都必须确保关键高信号内容对 Google 可抓取、可渲染、可稳定索引。

说明：

- 重点是结果，不是技术路线本身

### 2.6 AI Overview / 生成式搜索展示

不建议直接写：

> AI Overview 会优先展示我们的页面。

建议正式写法：

> 对相关实体和问题的完整覆盖，可能提升页面被生成式搜索结果引用的概率，但无法保证具体展示形式或展示位置。

说明：

- 当前生成式搜索结果仍存在较强的不确定性
- 正式 PRD 里应描述“可能性”和“条件”，不应写成展示承诺

## 3. 不建议直接写进 PRD 的表述

以下说法即使方向上有提醒价值，也不建议写进正式 PRD。

### 3.1 “Google 现在极度厌恶……”

原因：

- 语气太绝对
- 不适合作为正式项目文档措辞

建议替换：

> 当前搜索系统更倾向于奖励与搜索意图匹配、对用户真正有帮助的页面体验。

### 3.2 “这是核心排名信号”

原因：

- 如果没有 Google 官方明确确认，不建议写成直接 ranking signal

建议替换：

> 这通常与更好的搜索表现相关

### 3.3 “极高概率必然提升”

原因：

- SEO 有高度不确定性
- 正式 PRD 应描述目标、机制与验证方式，而不是写成保证结果

建议替换：

> 若执行到位并保持技术稳定，预期有较高概率改善主商业词的页面承接与搜索表现。

### 3.4 “一律不要 target=_blank”

原因：

- 这更像 UX 或产品规范，不是 Google 官方 SEO 红线

建议替换：

> 指向主转换页的关键站内链接应保持为可抓取、可传递上下文的普通站内链接，避免不必要的 `nofollow`。

## 4. 建议直接并入 PRD 的标准化句子

以下句子可以直接复制到正式 PRD 中。

### 4.1 Tool-first 总述

> 对于主词群对应的工具型搜索意图，主转换页必须首先满足“开始使用”需求，再满足“理解产品”需求。

### 4.2 首页边界

> 首页可以承担品牌认知、价值主张和入口引导，但不应与主转换页在完整功能流程上形成高度重叠。

### 4.3 主转换页边界

> 主转换页必须成为主词群的唯一核心承接页，并通过标题、首屏结构、内链和功能形态共同建立这一认知。

### 4.4 长尾实体覆盖

> 第一阶段通过主转换页区块和 FAQ 处理相关实体需求，以提升语义完整度，避免在页面数量上过早扩张。

### 4.5 可抓取性

> 所有高信号内容都必须在搜索引擎可抓取与可渲染的页面输出中稳定可见，而不是依赖用户交互后才出现。

### 4.6 KPI 定义

> 第一阶段的核心验证不是首页是否继续排名主词，而是目标 query cluster 是否逐步更多由主转换页承接。

补充示例：

> 示例验证方式：在 Google Search Console 的“查询”与“页面”维度中，观察主转换页在目标 query cluster 的 impressions 占比是否逐步提升。第一阶段可先以主转换页承接占比超过 60% 作为参考观察线，而不是绝对承诺值。

## 6. 建议结论

建议将外部评审反馈按本指南转写后，再并入正式 PRD 或评审纪要。

原因：

- 保留了高价值策略判断
- 避免把非公开算法细节写成硬规则
- 更适合产品、SEO、内容、前端跨团队协作

补充执行要求：

- 所有并入正式 PRD 的算法相关表述，均建议由 SEO 负责人或对应评审人做最终 wording 审核
- 特别避免出现“核心排名信号”“必然提升”“Google 明确厌恶”这类绝对化表述
